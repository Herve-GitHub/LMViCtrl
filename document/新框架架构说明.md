
**软件总体结构**
QtLvglDesigner 是一个“组态设计 + Lua 工程生成 + 桌面仿真 + 嵌入式 HMI 运行”的一体化工程。整体采用 Qt/qmake 多子工程组织，顶层工程在 QtLvglDesigner.pro 中按顺序包含 sdl2、LvglLuaBinding、simulator、designer 四个主要模块；Linux ARM HMI 运行端单独放在 linux_arm_dispaly 目录中，面向嵌入式设备构建。

```text
QtLvglDesigner
├─ designer/              组态设计器，Qt Widgets 图形界面
├─ designer/lua/          设计器随工程部署的 Lua 运行时与控件库
├─ LvglLuaBinding/        LVGL + Lua + 平台显示输入适配封装库
├─ simulator/             桌面仿真器命令行入口
├─ linux_arm_dispaly/     Linux ARM 嵌入式 HMI 启动程序
├─ sdl2/                  桌面仿真显示输入依赖
├─ bin/                   编译输出目录
└─ build/                 中间构建目录
```

**一、组态设计器模块**
设计器位于 designer，是用户进行 HMI 画面组态的主程序。它负责创建工程、管理图页、拖拽控件、编辑属性、保存工程，以及把工程编译为可运行的 Lua 脚本。

核心文件职责如下：

| 文件/模块 | 作用 |
|---|---|
| main.cpp | 设计器程序入口 |
| mainwindow.h、mainwindow.cpp、mainwindow_actions.cpp | 主窗口、菜单、工具栏、运行/编译/仿真等操作 |
| WidgetMeta.h | 定义工程、图页、控件实例、属性、事件、目标平台等核心数据结构 |
| projectmanager.h、projectmanager.cpp | 工程 JSON 保存/加载、Lua 编译、运行时资源部署 |
| luawidgetparser.h、luawidgetparser.cpp | 扫描 Lua 控件文件中的 `__widget_meta` 元数据，生成工具箱和属性面板数据 |
| canvasscene.h、canvasitem.h、canvasview.h | 画布、控件图元、拖拽缩放、撤销重做、对齐复制粘贴等编辑能力 |
| screentab.h | 单个图页的画布容器 |
| propertypaneldock.h | 控件属性编辑面板 |
| screenmanagerdock.h | 图页管理面板 |
| widgettoolbox.h | 控件工具箱 |
| lvglpreviewrenderer.h | 调用真实 LVGL 离屏渲染，生成控件预览 |

设计器保存的工程数据以 JSON 形式存在，通常命名为 `<projectName>.qlvgl.json`。编译后生成 `<projectName>.lua`，供仿真器和嵌入式 HMI 共同运行。

**二、Lua 运行时与控件库**
Lua 运行时位于 lua，这是设计器部署到每个工程目录中的运行资源。

```text
designer/lua
├─ runtime.lua                 工程运行时入口
├─ common/
│  ├─ common.lua               公共工具函数
│  └─ page_navigation.lua      页面跳转支持
└─ widgets/
   ├─ button.lua
   ├─ label.lua
   ├─ slider.lua
   ├─ chart.lua
   ├─ textarea.lua
   ├─ custom/valve.lua
   └─ ...
```

runtime.lua 的作用是加载设计器生成的 `project` 表，并按图页、层级、控件实例创建真实 LVGL 对象。它会扫描 `widgets/` 目录下所有 Lua 控件模块，根据每个模块中的 `__widget_meta.id` 建立控件注册表，然后调用控件模块的 `new(parent, state)` 创建控件实例。

运行时还提供了几个重要能力：

| 能力 | 说明 |
|---|---|
| 控件注册 | 自动扫描 `widgets/*.lua`，按控件 ID/type/文件名注册 |
| 图页构建 | 根据 `project.screens` 创建 LVGL screen |
| 层级控制 | 按 `zOrder` 升序创建控件，数值大的显示在上层 |
| 全局控件访问 | 控件按实例名注册到 `widgets.<Name>`，事件脚本可直接访问 |
| 事件绑定 | 读取 `on_xxx_handler` 等事件属性，动态编译 Lua 代码并绑定 |
| 页面跳转 | 创建全局 `PageManager`，支持 `goto_page()` / `select_page()` |
| 字体加载 | 支持工程字体配置，运行前设置默认字体 |

**三、LVGL Lua 绑定库**
绑定库位于 LvglLuaBinding，是整个系统的底层运行核心。它把 LVGL、Lua、平台显示输入、离屏渲染等能力封装为统一 C 接口，对上层设计器、仿真器、HMI 程序提供服务。

公共接口定义在 lvgl_lua_bindings.h，主要导出：

| 接口 | 用途 |
|---|---|
| `lvgl_lua_register()` | 向 Lua 注册 LVGL 控件、样式、事件等绑定函数 |
| `lvgl_simulator_run()` | 启动桌面仿真，创建 SDL 窗口并运行 Lua 工程 |
| `lvgl_hmi_run()` | 启动 Linux ARM HMI，使用 DRM/KMS 显示和 evdev 输入 |
| `lvgl_lua_render_widget_to_argb32()` | 设计器内离屏渲染控件预览 |
| `set_current_ttf_font()` / `get_current_ttf_font()` | 当前字体管理 |

绑定库中比较关键的实现文件：

| 文件 | 作用 |
|---|---|
| lvgl_lua_bindings.c | LVGL Lua 绑定注册总入口 |
| lvgl_obj_lua_bindings.c | 基础 LVGL 对象绑定 |
| lvgl_chart_lua_bindings.c | 图表控件绑定 |
| lvgl_slider_lua_bindings.c | 滑块控件绑定 |
| lvgl_textarea_lua_bindings.c | 文本输入控件绑定 |
| lvgl_lua_runtime.c | 创建 Lua 状态机、设置 `package.path`、运行 Lua 脚本 |
| lvgl_simulator.c | 桌面 SDL 仿真入口 |
| lvgl_hmi.c | Linux ARM DRM/evdev HMI 入口 |
| lvgl_offscreen_render.c | 设计器控件预览离屏渲染 |

**四、桌面仿真器**
仿真器位于 simulator，入口是 main.cpp。它本身保持轻量，只负责解析命令行参数，然后调用绑定库中的 `lvgl_simulator_run()`。

典型职责：

| 内容 | 说明 |
|---|---|
| 输入 | 设计器生成的 Lua 脚本路径 |
| 参数 | 窗口宽度、高度、标题 |
| 显示后端 | SDL 窗口 |
| 输入设备 | SDL 鼠标、键盘 |
| 执行方式 | 创建 LVGL 环境，创建 Lua 状态机，加载工程 Lua，进入 LVGL 主循环 |

运行形式示例：

```sh
QtLvglSimu project.lua --width 800 --height 480 --title DemoHMI
```

这部分用于客户现场演示时，可以说明为：桌面仿真器运行的就是设计器导出的 Lua 工程，与嵌入式端使用同一套控件逻辑和运行时数据结构。

**五、嵌入式 Linux ARM HMI**
嵌入式运行端位于 linux_arm_dispaly，入口是 main.cpp。它面向 Linux ARM 直连屏场景，调用绑定库中的 `lvgl_hmi_run()`，使用 DRM/KMS 输出画面，使用 evdev 接入触摸或按键输入。

主要参数：

| 参数 | 说明 |
|---|---|
| `<script.lua>` | 设计器生成的 Lua 工程入口 |
| `--drm` | DRM 显示设备，默认 `/dev/dri/card0` |
| `--connector` | DRM connector id，`-1` 表示自动选择 |
| `--touch` / `--pointer` | 触摸或鼠标 evdev 设备 |
| `--keypad` | 键盘或按键 evdev 设备 |
| `--log` | 日志输出文件 |

运行形式示例：

```sh
./hmi /path/to/project/project.lua --drm /dev/dri/card0 --connector -1 --touch /dev/input/event1
```

构建说明见 README.md。该模块与 Windows 桌面设计器分离，是 Linux/Unix 目标专用工程。

**六、核心运行流程**
整体软件链路可以概括为：

```text
1. 设计器启动
   ↓
2. 扫描 designer/lua/widgets 中的 Lua 控件元数据
   ↓
3. 用户在画布中拖拽控件、编辑属性、管理图页
   ↓
4. 工程保存为 <project>.qlvgl.json
   ↓
5. ProjectManager 编译生成 <project>.lua
   ↓
6. 部署 runtime.lua、common、widgets 到工程目录
   ↓
7. 桌面仿真器或嵌入式 HMI 加载同一个 Lua 工程
   ↓
8. Lua runtime 根据 project.screens 创建 LVGL 页面和控件
   ↓
9. LVGL 主循环刷新显示并处理触摸/键盘/事件脚本
```

**七、汇报时可强调的设计特点**
这个工程的核心亮点是“设计时”和“运行时”解耦：设计器负责可视化编辑和生成工程数据，运行端只需要加载 Lua 工程即可还原 HMI 界面。桌面仿真器和嵌入式 HMI 复用同一份 Lua 控件库、同一份工程脚本、同一套 LVGL Lua 绑定，只是底层显示输入后端不同：桌面端用 SDL，嵌入式端用 DRM/KMS 和 evdev。

这样做的好处是：客户在 PC 上看到的仿真效果，与设备端运行逻辑保持一致；控件扩展主要通过 Lua 模块和元数据完成；工程文件结构清晰，便于版本管理、部署和后续扩展。
